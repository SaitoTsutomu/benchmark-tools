# ベンチマークツール 改善計画書

2026年6月時点でのコードレビューに基づいた改善提案です。
主に「堅牢性の向上」「分析のしやすさ（観測可能性）」「実行効率」に焦点を当てています。

## 1. 堅牢性と信頼性の向上

### 1.1 リトライメカニズムの導入
LLM API は一時的なネットワークエラーやレートリミット (429) が発生しやすいため、単発の失敗でベンチマークを停止させず、リトライを行う仕組みを導入します。
- **提案**: `tenacity` ライブラリを導入し、`run_single_benchmark` に指数バックオフ（Exponential Backoff）によるリトライを実装する。
- **目的**: 大量項目実行時の完走率向上。

### 1.2 タイムアウトの明示的設定
API 応答が極端に遅い場合や、モデルが無限ループに近い挙動をした場合にプロセス全体が停止するリスクがあります。
- **提案**: `Runner.run_sync` または `LitellmModel` の設定に、最大タイムアウト（例: 60秒〜120秒）を明示的に設定する。
- **目的**: プロセスのハングアップ防止。

## 2. 観測可能性（Observability）の向上

### 2.1 エラー情報の永続化
現在、API エラーなどはログに出力されるのみで DB に保存されません。管理画面から失敗原因を特定できないため、これを改善します。
- **提案**: `Result` モデルに `error_message` (TextField) を追加し、例外発生時にその内容を保存する。
- **目的**: サーバーログを追わずに管理画面から失敗原因を特定できること。

### 2.2 判定プロセスの詳細保存
`check_judge` でコードを実行した際の「実際の出力」と「エラー内容」を保存することで、正解コードとの差分を容易に分析できるようにします。
- **提案**: `Result` モデルに以下を追加する。
    - `actual_output`: コード実行後の標準出力
    - `execution_error`: コード実行時の標準エラー（stderr）
- **目的**: プロンプト改善時のデバッグ速度向上。

### 2.3 トークン使用量の記録
コスト管理およびコンテキスト制限の分析のため、使用トークン数を記録します。
- **提案**: `Result` モデルに `prompt_tokens` と `completion_tokens` (IntegerField) を追加し、Litellm のレスポンスから取得して保存する。
- **目的**: モデルごとのコスト効率の測定。

---

## 3. 実行効率の最適化

### 3.1 Docker Sandbox の起動オーバーヘッド削減
現状、1つの項目につき最大2回 `docker run --rm` を行っています。項目数が増えた際にコンテナ起動時間が支配的になります。
- **提案**: 
    - 短期的: 必要に応じて `run_code` の呼び出し回数を最適化する。
    - 長期的: gVisor や Wasm 等のより軽量なサンドボックスの検討、または常駐コンテナへの stdin 入力方式への変更。
- **目的**: ベンチマーク実行時間の短縮。
